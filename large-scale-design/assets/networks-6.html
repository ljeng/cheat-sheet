The CAP Theorem

Consistency
Availability
Partition Tolerance

Consistency + Partition Tolerance

Traditional RDBMS approach: When partitioned, sacrifice availability to maintain consistency. System may become unavailable but data remains consistent.

Examples: Traditional SQL databases, HBase

Consistency + Availability

Theoretical only: Perfect consistency and availability, but cannot handle network partitions. Only works in perfect network conditions.

Examples: Single-node databases, systems with no distribution

Availability + Partition Tolerance

NoSQL approach: Remain available during partitions, but accept eventual consistency. System stays responsive but data may be temporarily inconsistent.

Examples: DynamoDB, Cassandra, CouchDB

Reality Check

The CAP theorem oversimplifies the tensions among properties. In distributed systems, partition tolerance is not optional — network faults will occur, and you must handle them.

Better Framing: "Consistent or Available When Partitioned"

⚠️ Network Partition Occurs ⚠️

Choose Consistency

Reject requests that might cause inconsistency

Result: System unavailable but data correct

Choose Availability

Accept requests and resolve conflicts later

Result: System responsive but potentially inconsistent

This reframing better captures the real-world tradeoff: when networks fail (not if), you must choose between maintaining strict consistency or keeping the system available to users.
